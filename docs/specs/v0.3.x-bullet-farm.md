# Spec: breadmin-composer v3 — Bullet Farm

## Overview

breadmin-composer v3 makes the impl-worker pipeline self-sustaining. Today the orchestrator
is batch-oriented: claim a fixed set of issues, dispatch agents in parallel, wait, then merge.
If an agent hangs or the orchestrator session dies, work stalls until a human intervenes.
Bullet Farm replaces that model with a **rolling dispatch loop** backed by an **agent watchdog**
and **crash-safe orchestrator recovery** — drawing on patterns from Gas Town's Witness and
continuous-feed Mayor, adapted to composer's GitHub-native, pipeline-structured model.

---

## Success Criteria

- [ ] `impl-worker` fills all available concurrency slots immediately at startup and refills
  them within 30 seconds of any agent completing, without human intervention
- [ ] An agent that exceeds `COMPOSER_AGENT_TIMEOUT_MINUTES` is automatically terminated,
  its issue is unassigned and re-queued, and its concurrency slot is refilled within one
  watchdog cycle (default: 60 s)
- [ ] If the orchestrator process is killed mid-pipeline and restarted, it resumes without
  human intervention: issues already assigned to open PRs are not re-dispatched; issues
  marked `in-progress` with no PR are automatically reclaimed and re-queued
- [ ] The orchestrator emits a heartbeat event to the conductor log every 60 seconds while
  running, recording active agent count, queue depth, and total cost to date
- [ ] `composer status` prints a live table of: active agents (issue #, branch, elapsed time),
  queue depth, completed this session, and total cost
- [ ] An agent that fails 3 times on the same issue posts a summary comment on the issue,
  removes `in-progress`, and does NOT block other issues from being dispatched

---

## Scope

**Included:**

- Rolling dispatch loop: orchestrator continuously refills concurrency slots as agents
  complete, rather than waiting for the entire batch to finish before starting the next
- Agent watchdog: background thread (or async task) that checks agent liveness on a
  configurable interval and terminates + requeues timed-out agents
- Orchestrator crash recovery: on startup, reconcile `in-progress` issues against open PRs
  to determine what is truly in-flight vs. orphaned; reclaim orphaned issues automatically
- Heartbeat logging: periodic conductor log events recording orchestrator vitals
- `composer status` subcommand: live snapshot of orchestrator state (readable from a
  second terminal while the worker runs)
- Per-issue failure isolation: one agent failing does not pause dispatch of other issues

---

## Constraints

- Must preserve all v2 security layers: explicit subprocess env dict, input sanitization,
  `--allowedTools` policy, `CLAUDECODE` nesting guard
- Rolling dispatch must respect the existing rate-limit concurrency caps
  (Pro: 1–2, Max: 2–3, Max 20x: 3–5) and the pre-dispatch budget check
- Watchdog and dispatch loop must not require a persistent daemon process; the orchestrator
  remains a foreground CLI command that can be started and stopped freely
- `composer status` must work by reading the conductor log and/or a small state file —
  no new IPC mechanism, no sockets
- No changes to the research-worker, design-worker, or plan-issues pipeline stages in this
  version; scope is impl-worker and orchestrator resilience only
- Checkpoint format must remain backward-compatible with v2 checkpoints (or provide a
  one-time migration path)

---

## Key Unknowns

1. What is the right mechanism for the watchdog to detect a hung agent — wall-clock timeout
   against the subprocess PID, absence of stdout events for N seconds, or a combination —
   and can the orchestrator reliably kill and clean up a Claude Code subprocess on macOS
   without leaving orphaned child processes or locked worktrees? The answer determines
   whether the watchdog runs as a thread, async task, or separate process.

2. When the orchestrator restarts after a crash, what is the reliable source of truth for
   distinguishing "agent completed and created PR" from "agent was mid-run with no PR"?
   Specifically: is `gh issue list --label in-progress` + `gh pr list` sufficient, or does
   the orchestrator need to maintain its own crash-safe state file (e.g., SQLite or
   atomic-write JSON) independent of GitHub? The answer determines the recovery architecture
   and whether a local state file is required.

---

## Non-Goals

- A persistent background daemon or system service (launchd, systemd)
- Changes to research-worker, design-worker, or plan-issues
- A web UI or TUI dashboard (beyond the `composer status` plaintext table)
- Multi-repo orchestration in a single session
- Gas Town's Beads ticketing system or any replacement for GitHub Issues as the work queue
- Automatic PR merging policy changes (orchestrator still merges after CI + review)
- Agent memory or context persistence across retries (each retry is a fresh subprocess)
