# breadmin-composer v0.3.x — Bullet Farm

**Theme:** Self-sustaining dispatch loop with watchdog and crash-safe recovery.

## Overview

Makes the implementation stage self-sustaining. Today the orchestrator is batch-oriented:
claim a fixed set of issues, dispatch agents in parallel, wait for the entire batch, then
merge. If an agent hangs or the orchestrator session dies, work stalls until a human
intervenes. Bullet Farm replaces that model with a **rolling dispatch loop**, an **agent
watchdog**, and **crash-safe recovery** — drawing on patterns from continuous-feed systems
adapted to composer's GitHub-native, pipeline-structured model.

---

## Success Criteria

- [ ] `composer run --impl --milestone <m>` fills all concurrency slots at startup and
  refills them within 30 seconds of any agent completing, without human intervention
- [ ] An agent exceeding `COMPOSER_AGENT_TIMEOUT_MINUTES` is automatically terminated;
  its issue is unclaimed and re-queued; its concurrency slot is refilled within one
  watchdog cycle (default: 60 s)
- [ ] An orchestrator process killed mid-pipeline restarts and resumes: issues with open
  PRs are not re-dispatched; issues with `in-progress` and no PR are reclaimed and re-queued
- [ ] The orchestrator emits a heartbeat log event every 60 seconds recording: active
  agent count, queue depth, and total cost to date
- [ ] `composer status` prints a live table: active agents (issue #, branch, elapsed time),
  queue depth, completed this session, total cost
- [ ] An agent that fails 3 times on the same issue posts a summary comment on the issue,
  removes `in-progress`, and does NOT block other issues from being dispatched

## Scope

**Included:**

- **Rolling dispatch loop**: continuously refills concurrency slots as agents complete;
  does not wait for the full batch before starting the next issue
- **Agent watchdog**: background thread that checks agent liveness on a configurable
  interval; terminates and re-queues timed-out agents
- **Orchestrator crash recovery**: on startup, reconcile `in-progress` issues against
  open PRs to distinguish in-flight vs. orphaned; reclaim orphaned issues automatically
- **Heartbeat logging**: periodic conductor log events recording orchestrator vitals
- **`composer status`**: live plaintext snapshot of orchestrator state readable from
  a second terminal while the worker runs
- **Per-issue failure isolation**: one agent failing does not pause dispatch of others
- **`composer run --qa`**: qa stage implementation (test suite + spec acceptance criteria
  check + bug filing loop)
- **`composer run --release`**: release stage implementation (tag + GitHub release + next
  version init)

## Failure Modes

**Agent watchdog:**
- Agent subprocess hangs with no stdout for N seconds → watchdog sends SIGTERM; if still
  alive after 5 s, sends SIGKILL; issue is unclaimed; slot is refilled
- Agent subprocess exits non-zero but produced a PR → orchestrator monitors PR as normal;
  exit code does not determine PR fate
- Watchdog thread itself crashes → orchestrator logs error, attempts restart; if watchdog
  cannot restart, halts dispatch and sends Telegram alert

**Rolling dispatch:**
- GitHub rate limit hit during dispatch → orchestrator backs off exponentially; does not
  drop issues from the queue; resumes when limit clears
- Concurrency slot count drops to zero (all agents timeout) → orchestrator logs warning,
  continues polling; does not terminate
- Issue claimed by another process between list and claim → orchestrator detects conflict
  on label write, skips issue, moves to next

**Crash recovery:**
- `in-progress` + open PR → resume monitoring PR
- `in-progress` + no PR → unclaim and re-queue
- `in-progress` + merged PR → close issue, free slot
- Checkpoint file missing → start fresh; log warning
- Multiple orchestrator instances detect the same orphaned issue → only one claims it
  (last-write-wins on GitHub label; the other sees the conflict and skips)

**`composer status`:**
- State file written by orchestrator is absent (orchestrator not running) → print
  "no active orchestrator session" and exit 0
- State file is stale (orchestrator crashed) → print last-known state with a staleness
  warning

## Interface Contracts

**`composer status` output:**

```
Active agents (2/5 slots):
  #42  feat/crash-recovery     3m 12s
  #51  feat/heartbeat          1m 04s

Queue: 3 issues pending
Completed this session: 7
Total cost: $1.24
```

**Heartbeat log event (conductor log JSONL):**

```json
{
  "event": "heartbeat",
  "run_id": "<uuid>",
  "ts": "<iso8601>",
  "active_agents": 2,
  "queue_depth": 3,
  "completed": 7,
  "cost_usd": 1.24
}
```

**Environment variables (new in v0.3.x):**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `COMPOSER_AGENT_TIMEOUT_MINUTES` | No | `60` | Max wall-clock time per agent before watchdog kills it |
| `COMPOSER_WATCHDOG_INTERVAL_SECONDS` | No | `60` | How often the watchdog checks agent liveness |

## Constraints

- Preserves all v0.2.x security layers: credential proxy, explicit subprocess env,
  `--allowedTools` policy, `CLAUDECODE` nesting guard
- Rolling dispatch respects existing rate-limit concurrency caps
  (Pro: 1–2, Max: 2–3, Max 20x: 3–5) and the pre-dispatch budget check
- Watchdog and dispatch loop do not require a persistent daemon; the orchestrator remains
  a foreground CLI command
- `composer status` works by reading a small state file written by the orchestrator —
  no new IPC mechanism, no sockets
- Checkpoint format remains backward-compatible with v0.2.x checkpoints (or provides
  a one-time migration path)
- No changes to the research, design, or scoping stages in this version

## Key Unknowns

1. **[P1]** Agent liveness detection: is wall-clock timeout against the subprocess PID
   sufficient, or does the watchdog need to monitor stdout event frequency (absence of
   stream-json for N seconds)? Can the orchestrator reliably kill a Claude Code subprocess
   on macOS without orphaned child processes or locked worktrees? The answer determines
   the watchdog mechanism and whether SIGTERM → SIGKILL sequencing is safe.

2. **[P1]** Crash recovery source of truth: is `gh issue list --label in-progress` +
   `gh pr list` sufficient to distinguish "completed with PR" from "mid-run no PR", or
   does the orchestrator need a crash-safe local state file independent of GitHub?
   The answer determines whether a local state store (atomic-write JSON or SQLite) is
   required for correct recovery.

---

## Non-Goals

- A persistent background daemon (launchd, systemd)
- Changes to research, design, or scoping stages
- A web UI or TUI dashboard beyond `composer status` plaintext
- Multi-repo orchestration in a single session (already in v0.2.x)
- Gas Town Beads ticketing or any replacement for GitHub Issues as the work queue
- Automatic PR merge policy changes
- Agent memory or context persistence across retries (each retry is a fresh subprocess)
