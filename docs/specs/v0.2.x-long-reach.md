# breadmin-composer v0.2.x — Long Reach

**Theme:** Extends to multi-model backends, CI deployment, multi-repo, and Telegram.

## Overview

Extends the v0.1.x pipeline to operate in more environments and with more backends.
Hardens the credential model so API keys never appear in subprocess env verbatim,
adds cross-model consensus for research quality, enables deployment in CI and on servers
without a personal Claude session, introduces Telegram progress notifications with
approval gates, and adds multi-repo orchestration from a single composer session.

---

## Success Criteria

- [ ] A research agent subprocess receives API credentials via a loopback credential proxy
  (not as plaintext env vars); intercepting the subprocess env reveals no raw API key
- [ ] `composer run --research --milestone <m>` completes successfully using Gemini CLI
  as an alternative backend when `COMPOSER_MODEL=gemini` is set
- [ ] `composer run --research --milestone <m>` completes successfully using GPT-4.1
  as an alternative backend when `COMPOSER_MODEL=gpt-4.1` is set
- [ ] A research doc produced by two different model backends is cross-validated; the
  orchestrator flags claims where models disagree for human review
- [ ] `composer run --all --milestone <m>` executes successfully when triggered by a
  GitHub Actions `issues` event, without a personal Claude Code session on the host
- [ ] The orchestrator sends a Telegram message at each stage transition
  (research complete, design complete, implementation complete, qa complete)
- [ ] The orchestrator sends a Telegram approval request before the release stage and
  waits for a human `/approve` reply before proceeding
- [ ] `composer run --all --milestone <m>` orchestrates implementation issues across
  two target repos simultaneously from a single composer session
- [ ] An orchestrator session killed mid-pipeline restarts and resumes without human
  intervention; no issues are double-dispatched and no `in-progress` labels are orphaned
- [ ] `make test` passes clean on the composer repo after all v0.2.x changes

## Scope

**Included:**

- **Credential proxy**: loopback SSE proxy that intercepts `ANTHROPIC_BASE_URL`; subprocesses
  receive a session token, not the raw API key; key rotation during long sessions
- **Multi-model backends**: Gemini CLI and GPT-4.1 support via a pluggable backend
  interface; backend selected by `COMPOSER_MODEL` config value
- **Cross-model consensus**: research docs produced by two backends are diff'd; disagreements
  are flagged in the doc with a `[DISPUTED]` marker for human review
- **CI/server deployment**: GHA workflow that triggers composer on `issues` events;
  deployment guide for running composer on a server without a personal session
- **Telegram integration**: stage-transition notifications and approval gate for release
- **Multi-repo orchestration**: single composer session dispatches impl agents to issues
  across multiple target repos; module isolation enforced per repo
- **Orchestrator crash recovery**: on restart, reconcile `in-progress` issues against
  open PRs; reclaim orphaned issues automatically

## Failure Modes

**Credential proxy:**
- Proxy process crashes mid-session → orchestrator detects missing proxy, restarts it
  before next dispatch; no agent is dispatched without a live proxy
- Key rotation fails → orchestrator logs error, pauses dispatch, sends Telegram alert;
  does not dispatch with a stale key
- Subprocess uses wrong `ANTHROPIC_BASE_URL` → proxy receives no requests; orchestrator
  detects no stream-json events and marks issue as failed

**Multi-model backends:**
- Backend binary not installed → `composer health` reports fatal; `composer run` exits
  with clear error before dispatch
- Backend returns non-zero exit → orchestrator retries up to `max_retries`; on exhaustion,
  unclaims issue and continues with remaining queue
- Backend produces no stream-json → orchestrator logs full stdout/stderr as diagnostic,
  marks issue failed, continues

**Telegram:**
- Telegram API unreachable → log warning, continue without notification; do not block
  pipeline progress on notification delivery
- Approval gate receives no response within timeout → orchestrator sends reminder after
  30 min; after 24 h without approval, sends final warning and halts (does not auto-release)

**Multi-repo orchestration:**
- One repo's CI fails → that repo's issues remain open; orchestrator continues with other
  repos; failing repo is reported in Telegram summary
- Module conflict across repos (same module claimed in two repos) → orchestrator serializes;
  does not dispatch second until first completes

**Crash recovery:**
- `in-progress` issue with open PR → resume monitoring; do not re-dispatch
- `in-progress` issue with no PR → unclaim and re-queue
- `in-progress` issue where PR is merged → close issue, continue
- Checkpoint file corrupt or missing → start fresh; log warning; do not inherit stale state

## Interface Contracts

**Environment variables:**

| Variable | Required | Purpose |
|----------|----------|---------|
| `COMPOSER_GH_TOKEN` | Yes | yeast-bot GitHub token (flows as `GH_TOKEN` to subprocesses) |
| `ANTHROPIC_API_KEY` | Yes (unless proxy) | Anthropic API key; replaced by proxy session token in subprocesses |
| `COMPOSER_MODEL` | No | Backend override: `claude` (default), `gemini`, `gpt-4.1` |
| `COMPOSER_TELEGRAM_TOKEN` | No | Telegram bot token; Telegram features disabled if absent |
| `COMPOSER_TELEGRAM_CHAT_ID` | No | Telegram chat ID for notifications |

**Telegram message format:**

```
[composer] <stage> complete: <version>
Repo: <owner>/<repo>
Issues closed: <N>
Duration: <HH:MM>
Next: <next stage>
```

Approval gate message:
```
[composer] Ready to release <version>. Reply /approve to proceed or /hold to pause.
```

## Constraints

- Raw API keys must never appear in subprocess env after the credential proxy is active
- Multi-model support must not change the research doc format or the triage rubric
- CI deployment must not require a persistent daemon; the GHA runner is ephemeral
- Telegram integration must degrade gracefully (no Telegram = pipeline still runs)
- Crash recovery must not require a separate state store beyond GitHub Issues + the
  existing checkpoint JSON; no SQLite or new IPC mechanism in this version
- All v0.1.x pipeline stages and CLI surface remain intact; no breaking changes

---

## Key Unknowns

1. **[P1]** Loopback SSE proxy: can `ANTHROPIC_BASE_URL` be set to a localhost endpoint
   that Claude Code subprocesses will respect without additional `--allowedTools` or
   network policy changes? The answer determines whether the credential proxy architecture
   is viable or requires a different isolation mechanism.

2. **[P1]** Gemini CLI tool allowlist: which tools must be explicitly allowed in the
   Gemini CLI's `--allowedTools` flag for research and impl agents to function correctly?
   The answer determines the dispatch command for Gemini-backend agents.

3. **[P2]** Cross-model consensus: what is the minimum viable diff strategy for comparing
   research claims across two model outputs — semantic similarity, structured claim
   extraction, or a third model as referee? The answer determines the consensus module design.

---

## Non-Goals

- A persistent background daemon (launchd, systemd)
- Rolling dispatch / agent watchdog (deferred to v0.3.x)
- `composer status` live dashboard (deferred to v0.3.x)
- A web UI or TUI
- Integration with any broker or financial data source
- Automatic PR merging policy changes
- Agent memory or context persistence across retries

## Pipeline Stages

### qa

After all implementation issues are closed:

1. Run full test suite: `uv run pytest`, `uv run ruff check`
2. Verify credential proxy: confirm raw API key does not appear in subprocess env
3. Exercise each Success Criterion against the running system
4. Verify Telegram notifications fire at each stage transition (using a test chat)
5. Verify approval gate blocks release until `/approve` received
6. File `stage/impl` + `bug` + `P0`/`P1` issues for any failures
7. Loop with implementation until qa runs clean

**qa acceptance criteria** (all must pass before release):
- All unit and integration tests pass
- No raw API keys visible in subprocess env during a monitored test run
- At least one end-to-end pipeline run completes with Gemini backend
- Telegram notifications delivered in a test chat for all stage transitions
- Crash recovery test: kill orchestrator mid-impl, restart, verify no double-dispatch

### release

1. Confirm zero open `P0`/`P1` issues on the milestone
2. `git tag v0.2.0 && git push origin v0.2.0`
3. `gh release create v0.2.0 --target mainline --generate-notes`
4. Send Telegram release notification
5. File `init: v0.3.0` pipeline issue
